# ADR-003: 알림 상태 설계

## 컨텍스트

알림 발송 흐름에서 시스템이 어떤 시점을 추적해야 하는지, 어떤 것을 상태로 관리하고 어떤 것을 별도 필드로 분리할지 결정이 필요하다.

## 결정

**발송 흐름에서 시스템이 추적해야 하는 시점을 기준으로 상태를 정의한다.**

```
접수 → (예약이면 대기) → 큐에 적재 → 발송 시도 → 성공 / 실패
```

이 흐름에서 추적이 필요한 시점:

- 접수됐지만 아직 큐에 들어가지 않은 시점 → **PENDING**
- 큐에 적재된 시점 → **QUEUED**
- 발송 성공 → **SENT**
- 재시도 소진 후 최종 실패 → **FAILED**

**예약 여부는 `scheduledAt` 필드로 분리한다.**

예약은 발송 흐름의 상태가 아니라 PENDING 진입 조건이다.
`scheduledAt` 값이 있으면 해당 시각이 됐을 때 큐에 넣고, 없으면 즉시 큐에 넣는다.

**읽음은 `readAt` 필드로 분리한다.**

읽음은 발송 흐름과 생명주기가 다르다. 발송 상태는 시스템이 관리하고, 읽음은 사용자 행동에 의해 바뀐다.
상태에 포함하면 발송 상태와 읽음 여부의 조합이 생겨 상태 머신이 아닌 플래그 조합이 된다.

**DELIVERED는 포함하지 않는다.**

도달 여부를 확인하려면 발송 인프라에서 콜백/웹훅으로 결과를 받아야 한다.
이메일은 사용할 인프라가 확정되지 않았고, 인앱도 도달 확인 메커니즘이 없으므로 흐름에 포함하지 않는다.

## 결과

- 상태 전이가 발송 흐름과 1:1로 대응되어 도메인 로직이 명확해짐
- 예약/읽음은 각각 독립적인 필드로 관리 가능
