# ADR-001: DDD + Hexagonal Architecture 채택

## 컨텍스트

알림 시스템은 컨텍스트 경계가 명확하고, 상태 전이, 재시도, 멱등성 등 비즈니스 규칙이 복잡하다.
메시지 큐, 발송 채널(이메일/인앱), 스케줄러를 과제에서는 내부 구현으로 대체하되,
운영 환경에서 실제 인프라로 교체 가능해야 한다.

## 대안

**Layered Architecture + 트랜잭션 스크립트**

Service 레이어에 비즈니스 규칙을 구현하는 방식.
interface를 사용하면 구현체 교체는 가능하지만, 교체 가능한 지점이 아키텍처 수준에서 명시적으로 드러나지 않는다.
비즈니스 규칙이 복잡해질수록 Service에 로직이 흩어지게 된다.

## 결정

DDD와 Hexagonal Architecture(Ports & Adapters)를 함께 채택한다.

**DDD** — 상태 전이, 재시도, 멱등성 등 비즈니스 규칙을 도메인 객체 안에 응집시킨다.
Service는 도메인 객체를 조합하는 역할만 하고, 규칙은 도메인이 책임진다.

**Hexagonal** — 교체가 예상되는 지점(메시지 큐, 발송 채널, 스케줄러)을 포트(interface)로 명시적으로 정의하고,
도메인이 인프라 구현을 모르게 한다. 인프라 어댑터가 도메인 포트를 구현하는 방향으로 의존성을 역전시킨다.

## 결과

- 비즈니스 규칙이 도메인 객체에 응집되어 Service가 단순해짐
- 교체 가능한 지점이 포트로 아키텍처에서 명확하게 드러남
- 도메인 테스트 시 포트를 Mock으로 교체 가능
- 패키지 구조가 Layered보다 복잡해지는 트레이드오프
