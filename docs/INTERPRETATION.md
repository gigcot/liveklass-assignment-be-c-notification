# 요구사항 해석 및 개선 의견

## 1. 재시도 구간 및 방식

요구사항에 "재시도 가능해야 함"만 명시되어 있고 어떤 구간에서 어떻게 재시도할지 정의되어 있지 않다.

발송 흐름의 각 구간별로 실패 가능성과 재시도 필요성을 판단했다.

| 구간 | 재시도 필요? | 이유 |
|---|---|---|
| 웹서버 (API 접수) | ❌ | 클라이언트가 재요청하면 됨 |
| DB 저장 | ❌ | DB 장애면 전체 장애 |
| MQ 발행 | ⚠️ 있으면 좋음 | MQ 일시 장애면 재시도로 복구 가능 |
| 발송 채널 | ✅ 필수 | 일시 장애가 자주 발생하고 재시도로 복구 가능 |

재시도가 가장 의미 있는 구간은 발송 채널이다.
알림은 실시간성보다 정확한 전달이 중요하고, 일시 장애는 잠깐 기다리면 복구되므로 횟수보다 간격이 중요하다.
발송 채널 실패에 대해 exponential backoff 방식으로 최대 3회 재시도하며, 간격은 4s → 8s → 16s(총 28s)로 설정한다.
재시도는 워커 내부에서 처리하며 재큐잉 없이 루프로 진행한다. 재시도 횟수는 서버(도메인)가 카운팅하고, 워커는 `/failed` 응답의 `retry` 플래그를 따른다. 워커에도 상한(4회)을 두어 서버 버그 시 무한루프를 방지한다.

MQ(Redis) 발행 실패도 일시 장애 시나리오가 있으므로 Relay에서 3회 exponential backoff(1s → 2s → 4s) 재시도를 수행한다. 재시도 소진 시 outbox_event를 PENDING으로 롤백하여 다음 폴링 사이클에서 재처리한다.

## 2. 중복 발송 방지 전략

요구사항에 "동일 이벤트에 대해 중복 발송 금지"가 명시되어 있으나 구체적인 방지 방법은 정의되어 있지 않다.

분산 환경에서 중복 발송이 발생할 수 있는 구간을 분석하고 레이어별로 대응한다.

| 구간 | 중복 원인 | 방지 방법 |
|---|---|---|
| API 접수 | 같은 이벤트로 요청이 두 번 들어옴 | `(userId, eventId, channel)` unique constraint |
| Relay | 여러 인스턴스가 같은 outbox_event 처리 | `SELECT FOR UPDATE SKIP LOCKED` |
| Worker | 큐에 중복 메시지, 혹은 여러 인스턴스 경합 | `/claim` — QUEUED→SENDING 원자적 전환 (SELECT FOR UPDATE) |

Worker의 `/claim`은 발송 직전 DB row를 SELECT FOR UPDATE로 잠근 후 QUEUED인 경우에만 SENDING으로 전환한다. 이미 SENDING/SENT/FAILED 상태면 409를 반환하고 해당 Worker는 발송을 skip하여 중복 발송을 방지한다.

분산 환경에서 exactly-once를 완벽히 보장하기 위해서는 발송 채널 자체의 멱등성도 필요하나, 채널 구현에 따라 다르므로 시스템 레이어에서 가능한 수준까지 방지하고 트레이드오프를 인지한다.

## 3. 응답 코드별 재시도 분기

요구사항에 재시도 조건이 명시되어 있지 않다.
발송 채널 응답에 따라 재시도 여부를 다르게 처리한다.

- **재시도**: 5xx (서버 일시 장애), 429 (레이트리밋), timeout (네트워크 문제)
- **즉시 FAILED**: 4xx (400, 401, 404 등) — 요청 자체가 잘못된 경우로 재시도해도 동일하게 실패

## 3. 알림 템플릿 관리 방식

요구사항에 "알림 템플릿 관리"라고만 명시되어 있고 방식이 정의되어 있지 않다.

운영 중에 템플릿 문구를 수정하기 위해 재배포하는 것은 현실적이지 않다.
템플릿을 API로 CRUD 관리하여 재배포 없이 운영 중에 변경 가능하도록 한다.

## 5. 조용히 죽은 알림 감지 및 처리

서버 장애나 MQ 유실로 인해 상태 전이가 중단된 알림이 PENDING, QUEUED, SENDING에 머무를 수 있다.

- **PENDING**: 접수 후 Relay가 미처 적재하기 전에 장애 발생
- **QUEUED**: Redis에 적재됐지만 Worker가 처리 전에 장애 발생 (outbox는 RELAYED)
- **SENDING**: Worker가 `/claim` 후 처리 도중 크래시

이런 알림은 재시도 로직에 진입하지 못한 채 영원히 남아 있게 된다.

스케줄러가 주기적으로 PENDING/QUEUED/SENDING 상태의 알림 중 `updatedAt` 기준으로 일정 시간이 지난 것을 스캔해서 FAILED로 전환한다.

발송 완료 시각은 `sentAt` 필드로 별도 관리한다.
`updatedAt`은 `markRead()` 등 다른 이벤트로도 덮어써지므로 "언제 발송됐는지"를 정확히 추적하려면 `markSent()` 시점에만 세팅하는 전용 필드가 필요하다.
`sentAt`은 `markSent()` 시점에만 세팅하며, "실제로 채널에 발송된 시각"을 정확히 추적하는 데 사용한다.

구체적인 타임아웃 기준은 발송 채널 특성에 따라 조정이 필요하다.

## 6. 알림 객체의 templateId 참조

알림 타입과 채널은 템플릿이 이미 정의하고 있으므로 Notification이 type, channel을 별도로 저장하는 것은 중복이다.
Notification은 templateId만 참조하고, 타입/채널 정보는 템플릿 테이블에서 확인한다.

접수 시점에 templateId를 고정하므로 이후 템플릿이 수정되더라도 이 알림은 접수 당시 버전으로 발송된다.
별도 템플릿을 지정하지 않으면 해당 타입/채널의 최신 템플릿을 사용한다.

## 4. referenceData 구조

알림 타입별로 참조 데이터 구조를 강제하면, 알림 타입 추가 시마다 도메인 코드 변경이 필요하다.
알림 타입은 계속 추가되는 것이므로 이는 비현실적이다.

referenceData를 Map으로 유연하게 받고, 발송 시점에 템플릿 플레이스홀더에 필요한 값이 있는지 검증하는 방식으로 처리한다.
알림 타입 추가 시 도메인 코드 변경 없이 템플릿만 추가하면 된다.
