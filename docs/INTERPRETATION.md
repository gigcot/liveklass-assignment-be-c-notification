# 요구사항 해석 및 개선 의견

## 1. 재시도 구간 및 방식

요구사항에 "재시도 가능해야 함"만 명시되어 있고 어떤 구간에서 어떻게 재시도할지 정의되어 있지 않다.

발송 흐름의 각 구간별로 실패 가능성과 재시도 필요성을 판단했다.

| 구간 | 재시도 필요? | 이유 |
|---|---|---|
| 웹서버 (API 접수) | ❌ | 클라이언트가 재요청하면 됨 |
| DB 저장 | ❌ | DB 장애면 전체 장애 |
| MQ 발행 | ⚠️ 있으면 좋음 | MQ 일시 장애면 재시도로 복구 가능 |
| 발송 채널 | ✅ 필수 | 일시 장애가 자주 발생하고 재시도로 복구 가능 |

재시도가 가장 의미 있는 구간은 발송 채널이다.
알림은 실시간성보다 정확한 전달이 중요하고, 일시 장애는 잠깐 기다리면 복구되므로 횟수보다 간격이 중요하다.
발송 채널 실패에 대해 exponential backoff 방식으로 3회 재시도한다.
구체적인 재시도 간격은 발송 채널 확정 후 해당 채널의 특성에 맞게 조정이 필요하다.

MQ 발행 실패도 일시 장애 시나리오가 있으므로 재시도를 고려한다.

## 2. 응답 코드별 재시도 분기

요구사항에 재시도 조건이 명시되어 있지 않다.
발송 채널 응답에 따라 재시도 여부를 다르게 처리한다.

- **재시도**: 5xx (서버 일시 장애), 429 (레이트리밋), timeout (네트워크 문제)
- **즉시 FAILED**: 4xx (400, 401, 404 등) — 요청 자체가 잘못된 경우로 재시도해도 동일하게 실패

## 3. 알림 템플릿 관리 방식

요구사항에 "알림 템플릿 관리"라고만 명시되어 있고 방식이 정의되어 있지 않다.

운영 중에 템플릿 문구를 수정하기 위해 재배포하는 것은 현실적이지 않다.
템플릿을 API로 CRUD 관리하여 재배포 없이 운영 중에 변경 가능하도록 한다.

## 5. 조용히 죽은 알림 감지 및 처리

서버 장애나 MQ 유실로 인해 상태 전이가 중단된 알림이 PENDING 또는 QUEUED에 머무를 수 있다.
이런 알림은 재시도 로직에 진입하지 못한 채 영원히 남아 있게 된다.

스케줄러가 주기적으로 PENDING/QUEUED 상태의 알림 중 `createdAt` 기준으로 일정 시간이 지난 것을 스캔해서 FAILED로 전환한다.

발송 완료 시각은 `sentAt` 필드로 별도 관리한다.
`updatedAt`은 `markRead()` 등 다른 이벤트로도 덮어써지므로 "언제 발송됐는지"를 정확히 추적하려면 `markSent()` 시점에만 세팅하는 전용 필드가 필요하다.
`sentAt`은 `markSent()` 시점에만 세팅하며, "실제로 채널에 발송된 시각"을 정확히 추적하는 데 사용한다.

구체적인 타임아웃 기준은 발송 채널 특성에 따라 조정이 필요하다.

## 6. 알림 객체의 templateId 참조

알림 타입과 채널은 템플릿이 이미 정의하고 있으므로 Notification이 type, channel을 별도로 저장하는 것은 중복이다.
Notification은 templateId만 참조하고, 타입/채널 정보는 템플릿 테이블에서 확인한다.

접수 시점에 templateId를 고정하므로 이후 템플릿이 수정되더라도 이 알림은 접수 당시 버전으로 발송된다.
별도 템플릿을 지정하지 않으면 해당 타입/채널의 최신 템플릿을 사용한다.

## 4. referenceData 구조

알림 타입별로 참조 데이터 구조를 강제하면, 알림 타입 추가 시마다 도메인 코드 변경이 필요하다.
알림 타입은 계속 추가되는 것이므로 이는 비현실적이다.

referenceData를 Map으로 유연하게 받고, 발송 시점에 템플릿 플레이스홀더에 필요한 값이 있는지 검증하는 방식으로 처리한다.
알림 타입 추가 시 도메인 코드 변경 없이 템플릿만 추가하면 된다.
